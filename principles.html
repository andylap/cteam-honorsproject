Object-Oriented Programming Principles

Overview

Object-Oriented Programming is a paradigm that most, if not all, developers first learn when starting their journey into programming. While it's not perfect, it's a tried and tested approach to writing a clean, well structured, and highly reusable codebase. That's what all programming paradigms and approaches aim to do: reduce code for the ease of expansion and maintenance, ensure separation of concerns, and ensure code reusability.

When it comes to object-oriented programming, there are 4 main principles:

  1. Encapsulation
  2. Abstraction
  3. Inheritance
  4. Polymorphism

Let's dive in and find out what all of these mean! Along the way, we'll be using the features of cars and trucks to help illustrate these principles.

Encapsulation

The whole point of encapsulation is to privatize the state and to publicize methods. What the heck does that mean, you might ask? Let's take the car example. If you were a car manufacturer, you wouldn't want the end-customer to have direct functionality to mutate (change) or access the odometer. Instead, you'd only want that information to be read-only. As a result, you might create a button on the navigation display that will then display a pop-up with the number of miles driven. Similarly, you wouldn't ever just let them change the speedometer. Instead, you would provide methods (or ways) for these states (or properties) to be mutated. Let's look at an example.

Step 1: Define Private State Properties

First, let's define some (private) state variables for all cars. These properties will access the values associated with important car properties. For example, these class properties might define miles driven, miles driven, speed of travel, the gear currently engaged, last inspection date, etc. By holding these properties in private state variables, we have a specific and known location where we can always access these critical values. As such, we want them to be private to ensure that no one accidentally (or intentionally) changes these values in an unexpected or harmful way.

Step 2: Define Private Methods

Now let's define some of the private methods that we'll invoke occasionally. Notice that these are private. This means that they can only be invoked within the class. This is because we don't want to grant anyone else the ability to call these methods. This is generally done to prevent errors. It helps us ensure that these methods aren't being called at unexpected times or being forgotten.

Step 3: Define Public Methods

Lastly, let's define some publicly accessible methods that will let others use our car. We'll make these public because we want others to be able to update the private state variables in a controlled and expected manner.

TL;DR

By specifying the access level of the properties and methods of a class, we're able to control the ways in which others interact with our class's data. As a result, we'll hide the values themselves and expose ways to retrieve and update those values through public methods.

Abstraction

Abstraction lets us hide the implementation details of our code to outsiders. Sometimes you'll hear others refer to this as the "black box" principle. This essentially exposes a method to others without explaining how it's written.

Still, confused? Think about your TV. Instead of telling you all the steps needed to produce video, the manufacturer provides a remote with a button (or method) to turn the TV on and off. This ensures that the user doesn't need to invoke all of the necessary subprocesses, let alone in the right order, to complete the task of turning the TV on and off.

Let's drive this point home with the Car class example again.

First, we'll define in our method just some of these processes that might occur when you fill up your gas tank. Perhaps while you're reading this, you're realizing things that might happen behind the scenes that you never thought of. That's exactly why abstraction is great- it takes away the need to worry about the underlying details that make everything go smoothly.

Now that we've defined our method, let's call it. Notice that it's just a simple call on our Car class instance. With just that simple line we're able to call all of the subprocesses that together let us fill up the tank without error every time. Also notice that we don't let others invoke only a subroutine. Congrats, now you know what abstraction is!

Inheritance

Inheritance is the heart and soul of OOP. Essentially this principle allows us to reuse code among related classes. It gives us a way to structure code hierarchically. Additionally, it allows us to indicate relationships among classes, such as parents, children, and sibling classes. What the heck does all this mean? Let's go straight into the code to get a better understanding.

Step 1: Define a Parent Class

First, we'll define a parent class. This will be the universal class that all subclasses (or child classes) will have direct and automatic access to. In our previous examples, we've classified our properties and methods as features of a car. However, now we're looking at the bigger picture and realize that these are properties of a vehicle (i.e. it can apply to a motorcycle and truck, too). So let's adjust our code to define a generic vehicle instead. Notice that all of the properties and methods have definitions.

[Optional] Step 2: Declare an Interface (or Protocol)

Now let's define any interfaces (or protocols) that might apply to vehicles. Why an interface? Let's take the example of an electric car. If we were to define the "layers" of an electric vehicle, we'll quickly identify that it's a vehicle. However, our Vehicle parent class does not mention anything about being electric capabilities. This helps us ensure the best reusability without adding features irrelevant to some child classes. Instead, we'll declare an ElectricCapable interface that can be inherited by only vehicles with electric capabilities. Instead of defining how the extra properties and methods of electric car work, we'll just indicate (with an interface) that we want those properties and methods defined, but we'll leave it up to the inheriting class to do the work for us.

Step 3: Create a Subclass

Now, let's define a child class that inherits a parent class. For this example, let's define a Tesla class. This will allow us to inherit the Vehicle class. Additionally, given the electric capabilities of a Tesla, we'll inherit the ElectricCapable interface (or protocol). That will help us add the additional features that a generic car doesn't always have. You'll notice that none of the vehicle methods or properties are in the Tesla class. That's because all of the non-private properties and methods are automatically part of all Tesla instances. As a result, we've saved ourselves a ton of work and created very reusable code in the Vehicle class.

Polymorphism

Polymorphism in Greek translates to "many forms." Given this, we can start thinking about how that idea might apply to OOP. Let's think about our vehicle example in the context of cars and trucks. More specifically, let's take a look at a vehicle's ability to accelerate. From experience, you probably know that cars are able to accelerate faster than trucks. Now let's keep this information in mind when defining our Vehicle subclasses. Knowing that our cars and trucks will accelerate at different speeds, do we want to define in our Vehicle class the speed at which all vehicles will accelerate? Instead, why don't we define an interface called Driveable. When a class inherits Driveable (i.e. a car, motorcycle, or truck), we'll let that subclass define the accelerate function. This will ensure that each different type of vehicle updates the speed property of a car correctly.
