Overview

This section will give some background information regarding compilers/interpreters for different languages and platforms. It will also provide a brief overview of how to choose and install certain more popular compilers.

Compiled vs. Interpreted Languages

First, a little background information.

Programming langauges are typically classified as either compiled or interpreted based on the process needed to run a program. Although the difference between compiled and interpreted languages can get blurry with all the different compilers/interpreters out there, it's a good starting point to understand how code actually runs.

Programs written in compiled languages are executed directly on the computer’s processer where a compiler turns the code into machine language that the processor can understand. Then, at runtime, the machine language is used to execute your program. Compiled languages require a two step process to run a program.

Interpreted languages, on the other hand, are programming languages that are not machine code before runtime (no separate compilation step). Therefore, the code is translated to machine language at runtime when the code is being executed, potentially slowing down the process. The easiest way to differentiate it from a compiled langauge is that running a program written in an interpreted language is normally a one step process.

Is C++ compiled or interpreted?

C++ is a traditional compiled langauge, in that it's a two step process where .o files containing machine code are created from source code. Then, those .o files are compiled into an executable that can be run by the user. C++ is also a language where the compilation process is platform dependent. The .o files generated on a Windows system cannot be used by a Linux or Mac system, and vice versa.

Java is a little different than C++, as it is a language that is neither compiled nor interpreted. (Alternatively, some say it's both compiled and interpreted). In C or C++, the executable files generated after compilation can only be run on that specific operating system (a C++ program compiled on a Mac can only run on a Mac).

Java, however, takes a different approach. Java designers reorganized the compile-link-execute sequence used in C/C++, and developed low level means of performing that sequence with platform independence.

While a C/C++ compiler creates an object file, the Java compiler creates a bytecode file, which is an object file for the virtual machine (often referred to as the Java Virtual Machine or JVM).

The reason why Java isn't classified as strictly compiled or interpreted is partially because of how dependent it is on the execution environment. In most cases, JVMs use a technique called Just-in-Time (JIT) compilation to compile the bytecode to machine code understood by the processor at runtime - a mix between strict compilation or interpretation. Additionally it is technically possible to run Java as strictly interpreted or compiled. Now, we can see how blurred the lines can get between compiled and interpreted langauges with modern execution environments.

To summarize, the high level steps for compiling and running a program in Java work like this:

  1. Write source code
  2. Compile source code to .class files (platform independent)
  3. Execute .class files with JVM (platform specific)

To develop in Java, it's highly reccomended to use an IDE (Integrated Development Environment). Some of the more popular IDEs are explored on our Setup page. All the options can be a little overwhelming but once you choose one and commit to using it, it'll become second nature in no time.

Is Python interpreted or compiled?

Answer: It depends. Python is officially classified as an interpreted language, as it generates bytecode (machine langauge) from your source code and and executes the bytecode in one step.

The process of compiling and linking in python is similar to Java. Source code is saved as a .py file, and then that is compiled to bytecode like Java (platform independent). Bytecode is stored in .pyc files that can be run by the PVM (Python Virtual Machine).

If you're curious about where the line is between interpreted and compiled langauges, the Python documenation explains it like this:

"Python is an interpreted language, as opposed to a compiled one, though the distinction can be blurry because of the presence of the bytecode compiler. This means that source files can be run directly without explicitly creating an executable which is then run."

Finally, there are several interesting Python compilers and interpreters designed to work with other languages, and a few will be discussed below.

CPython is the default interpretation that is installed when you download Python from python.org . Written in C, CPython follows the traditional formal of compiling source into bytecode and interpreting that bytecode.

PyPy is heralded as a faster Python interpreter - in fact, it is on average 4.3x faster than CPython. PyPy maintains its speed by using JIT (Just-in-Time) compilation and other behind the scenes optimization. In most cases, you're able to swap out CPython for PyPy without significant changes being required.

Brython is a Python compiler that converts Python code into JavaScript code. Brython is built for the HTML5 environment and has a built in interface to the DOM (Document Object Model) objects and events. It comes prepackaged with relevant libraries, suppports almost all features of CPython, and provides support for all modern browsers.

WinPython is a Python distribution that is specifically created for the Windows operating system. Earlier versions of CPython weren’t designed for Windows resulting in bugs. While CPython is now very stable on Windows it still has a few benefits. It comes prepackaged with some of the more popular Data Science and Machine Learning libraries (NumPy, SciPy, Pandas, etc.) and bundled features like a C and C++ compiler.

This is all a little overwhelming. What about all of these online compilers? Can I just use those?

Online compilers are fantastic for simple debugging and testing, but it's best to start using and familiarlizing yourself with creating a local development environment. Virtually all reasonably sized projects - for school, hobby, and work - require familiarity with a local development environment. With all the options out there, choosing what to use (especially on Windows) can be quite overwhelming, but hopefully this page gives an idea of some of the options out there.

And - speaking from a Windows perspective - setting up a development environment within a Linux Subsystem can be much easier that delving into the world of Windows development. This is especially true if collaborating with someone working on a MacOS or Linux system.
