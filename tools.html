Compiled vs. Interpreted Languages

First, a little background information.

Programming langauges are typically classified as either compiled or interpreted based on the process needed to run a program. Although the difference between compiled and interpreted languages can get blurry with all the different compilers/interpreters out there, it's a good starting point to understand how code actually runs.

Programs written in compiled languages are executed directly on the computer’s processer where a compiler turns the code into machine language that the processor can understand. Then, at runtime, the machine language is used to execute your program. Compiled languages require a two step process to run a program.

Interpreted languages, on the other hand, are programming languages that are not machine code before runtime (no separate compilation step). Therefore, the code is translated to machine language at runtime when the code is being executed, potentially slowing down the process. The easiest way to differentiate it from a compiled langauge is that running a program written in an interpreted language is normally a one step process.

Is C++ compiled or interpreted?

C++ is a traditional compiled langauge, in that it's a two step process where .o files containing machine code are created from source code. Then, those .o files are compiled into an executable that can be run by the user. C++ is also a language where the compilation process is platform dependent. The .o files generated on a Windows system cannot be used by a Linux or Mac system, and vice versa.

Java is a little different than C++, as it is a language that is neither compiled nor interpreted. (Alternatively, some say it's both compiled and interpreted). In C or C++, the executable files generated after compilation can only be run on that specific operating system (a C++ program compiled on a Mac can only run on a Mac).

Java, however, takes a different approach. Java designers reorganized the compile-link-execute sequence used in C/C++, and developed low level means of performing that sequence with platform independence.

While a C/C++ compiler creates an object file, the Java compiler creates a bytecode file, which is an object file for the virtual machine (often referred to as the Java Virtual Machine or JVM).

The reason why Java isn't classified as strictly compiled or interpreted is partially because of how dependent it is on the execution environment. In most cases, JVMs use a technique called Just-in-Time (JIT) compilation to compile the bytecode to machine code understood by the processor at runtime—a mix between strict compilation or interpretation. Additionally it is technically possible to run Java as strictly interpreted or compiled. Now, we can see how blurred the lines can get between compiled and interpreted langauges with modern execution environments.

To summarize, the high level steps for compiling and running a program in Java work like this:

  1. Write source code
  2. Compile source code to .class files (platform independent)
  3. Execute .class files with JVM (platform specific)

To develop in Java, it's highly reccomended to use an IDE (Integrated Development Environment). Some of the more popular IDEs are explored on our Setup page. All the options can be a little overwhelming but once you choose one and commit to using it, it'll become second nature in no time.

Is Python interpreted or compiled?

Answer: It depends. Python is officially classified as an interpreted language, as it generates bytecode (machine langauge) from your source code and and executes the bytecode in one step.

The process of compiling and linking in python is similar to Java. Source code is saved as a .py file, and then that is compiled to bytecode like Java (platform independent). Bytecode is stored in .pyc files that can be run by the PVM (Python Virtual Machine).

If you're curious about where the line is between interpreted and compiled langauges, the Python documenation explains it like this:

"Python is an interpreted language, as opposed to a compiled one, though the distinction can be blurry because of the presence of the bytecode compiler. This means that source files can be run directly without explicitly creating an executable which is then run."

Finally, there are several interesting Python compilers and interpreters designed to work with other languages, and a few will be discussed below.

CPython is the default interpretation that is installed when you download Python from python.org . Written in C, CPython follows the traditional formal of compiling source into bytecode and interpreting that bytecode.

PyPy is heralded as a faster Python interpreter—in fact, it is on average 4.3x faster than CPython. PyPy maintains its speed by using JIT (Just-in-Time) compilation and other behind the scenes optimization. In most cases, you're able to swap out CPython for PyPy without significant changes being required.

Brython is a Python compiler that converts Python code into JavaScript code. Brython is built for the HTML5 environment and has a built in interface to the DOM (Document Object Model) objects and events. It comes prepackaged with relevant libraries, suppports almost all features of CPython, and provides support for all modern browsers.

WinPython is a Python distribution that is specifically created for the Windows operating system. Earlier versions of CPython weren’t designed for Windows resulting in bugs. While CPython is now very stable on Windows it still has a few benefits. It comes prepackaged with some of the more popular Data Science and Machine Learning libraries (NumPy, SciPy, Pandas, etc.) and bundled features like a C and C++ compiler.

A few other interesting Python options can be found here (Compilers) or here (Interpreters) .

Bash Commands

Whether you are working on an individual project or in a large team setting, knowing how to use the Bash terminal is useful in a variety of settings. The commands you will be learning to work with will allow you to efficiently navigate your local file system; manipulate, move, and display file contents and statistics; and interact with file permissions management.

File System Navigation

There are several general functions which are incredibly useful in allowing you to navigate the file system. Some of the basics include:

When working in the local file system, many commands often rely on the locations of the files or directories being worked with as input arguments to the commands being used. This allows the commands being used to manipulate documents very precisely. To properly pass document locations to commands, there are two positioning systems that can be used: Relative, and Absolute.

Relative Position

Relative positioning is when locations are specified relative to the users current location in the file system.

There are several symbols which are used to specify locations relative to the current position

Symbol Purpose
. Represents the current location in the file system. This is useful when specifying destinations.
Example: moving and copying documents.
.. This marks the parent directory from the current position in the file system. To specify multiple directories back, the symbol can be strung together using /.
Example: ../../..
* The asterisk denotes all files in the current directory. This may be useful when trying to move many files from one directory to another.

Absolute Position

Absolute positioning is when locations are specified relative to the base directory in the file system. Using absolute position may be beneficial when working with two or more directories that are not closely positioned in the file system. Rather than using extensive relative movement, it may be easier to track which directories are being utilized with absolute position. 

$ pwd

Display the absolute position of your current location in the file system.


Example:

/mnt/c/Users/Name/Folder1/Folder2 
$ ls
code/ HelloWorld.txt overview.txt

List all of the folders and files in the current directory. Any item ending with / is a folder, and the other items are files.

$ ls -l
total 0
drwxrwxrwx 1 USER GROUP 4096 Nov 29 17:26 code/
-rwxrwxrwx 1 USER GROUP   14 Nov 29 17:10 HelloWorld.txt
-rwxrwxrwx 1 USER GROUP  516 Nov 29 17:09 overview.txt

The following is a breakdown of the output:

  1. Field 1 (-/d): marks whether the item is a file (-) or directory (d)
  2. Field 2-4 (rwx): marks the permissions that file owner has, followed by the group permissions, and ending with the general file permissions to apply to everyone who has the file.
    • r: read
    • w: write
    • x: execute
  3. Field 5: 1
  4. Field 6: The owner of the file
  5. Field 7: The group the file belongs to
  6. Field 8 (Number): The size, in bytes, of the file or directory.
  7. Field 9: The date the file was last modified
  8. Field 10: The name of the file or directory

It should also be noted that you can pass in a relative or absolute position to display the contents of other directories.

Examples:

$ ls ../

Display the contents of the parent folder to your current location in the file system.

$ ls -l /mnt/c/Users/Name/Folder1/

Display the contents of Folder1

$ cd \argument

cd allows you to change the directory you are working in by switching to a relative or absolute location in the file system.

Examples:

Exit the current folder, enter differentFolder, and proceed to enter Folder2 from there. Set this new location as the working directory.

$ pwd
/mnt/c/Users/Name/differentFolder/Folder2

OR:

$ cd /mnt/c/Users/Name/differentFolder/Folder2/

Set the working directory to Folder2 from the absolute path.

$ pwd

$ mv fileToMove folderToMove/ ... destination/

Files and Command Chaining

The following commands will walk you through how to create and remove files and folders as well test program outputs, chain commands together, and alter file permissions.

Touch allows users to quickly create files which can be opened in the terminal and altered. Multiple files can be created using one instance of the command by stringing the desired file names together. It is important to note that the generated files will, however, be completely empty. 

$ touch filename.txt file2name.data
    .
    ├── file2name.data
    └── filename.txt

Relative and Absolute positions can be passed to the command to create the files in specific locations without having to change the working directory.

mkdir works similarly to touch, with the difference being that mkdir generates empty directories, rather than empty files. The command accepts the same input parameters as touch, and folder names can be strung together to create multiple folders using one instance of the command.

$ mkdir example1/ example1/example2/
    .
    ├── example1
    │   └── example2
    ├── file2name.data
    └── filename.txt

The input arguments can also be strung together in ways that create folders inside of folders that were just created. The mkdir command creates the folders in the order they are specified, so switching the order of the arguments of the example above would not work because the example1 folder would not have been created before trying to generate example1/example2/.

The rm command specifies the removal of specific files in a directory. There are additional flags/arguments you can use to specify the removal of entire directories, or specify that all files in a directory should be removed without removing the directory itself.

Examples:

The following is the file structure that will be used for these examples. The file structure will reset after removal command.

    .
    ├── example1
    │   ├── example2
    │   │   └── file2.txt
    │   └── file1.txt
    ├── file2name.data
    └── filename.txt

The following command will remove the specified file from the working directory.

$ rm filename.txt file2name.data
    .
    └── example1
        ├── example2
        │   └── file2.txt
        └── file1.txt

Additionally, the user can use the * to specify all files that relate to a certain condition. In the following example that condition is to remove all files which names start with file

$ rm file* 
    .
    └── example1
        ├── example2
        │   └── file2.txt
        └── file1.txt

The following are additional cases of how * can be used to specify specific files.

Symbol What it Removes
*.txt Removes all files in the working directory that end with a .txt extension
file*name.txt Removes all text files which names start with file and end with name.txt.
* Removes all files, but does not alter directories.
-r * Removes everything from the working directory, including all directories and subdirectories.

An additional argument that should be learned is the -r flag. This specifies that the remove command should recursively remove files. This is useful if you are trying to remove directories that contain files inside them. Rather than having to specify all of the files in each subdirectory, the user can specify that all contents of a directory should be removed recursively.

$ rm -r example1/

results in a file structure of:

    .
    ├── file2name.data
    └── filename.txt
$ more filename

more is used to display the contents of a file. This is useful if you are trying to determine which file has specific contents. Rather than having to open each file in an editor to find what you are looking for, you can use more to quickly see what each file contains.

Similarly to the more command, head and tail can be used to display the top (head) or bottom (tail) of a file. The number of lines to display can be specified using the -n argument. It should be noted that the number of lines does not specify the number of sentences to display, so in a text document, head and tail can cut off sentences.

$ head -n2 overview.txt

The end of the overview document displayed in these examples is a blank line.

When looking for specific statistical information regarding a file and its contents, wc can be a useful first command to explore. Calling wc on a file will display:

  1. The file's line count
  2. The file's word count
  3. The number of characters in the text of the file
$ wc overview.txt

Setting user file permissions may be important if you want to make sure that a file is secure, cannot be changed, and/or cannot be run without your knowledge. chmod allows users to specify who has access to, the ability to alter, and the ability to execute specific files in the file system. And these permissions follow the file in the event that the file is transferred from one computer to another. It is even possible to revoke permission from your own computer.

There are a few things that need to make sense before chmod will make sense. There are 3 permissions that are able to be altered: read (4), write (2), and execute (1). Additionally, there are 3 groups for which permissions can be set: user (first number), group (second number), and non-group users (third number). In order to set the permissions for each group, you will run chmod followed by 3 numbers. The first number for the user, the second number for the group, and the third number for the non-group user. And the number that each of these permissions should be, is the sum of the permissions that you want to give each group.

Examples:

To give read, write, and execute access: 4 (read) + 2 (write) + 1 (execute) = 7

To give only read and execute access: 4 (read) + 1 (execute) = 5

Now to set the permissions of a file for all three types of user.

$ chmod 754 overview.txt

The && and || operators are useful for conditionally chaining commands together. Linking together two commands with && will run command1, then continue onto command2 only if command1 successfully completes. And linking together two commands with || will run command1, then continue onto command2 whether command1 successfully completes or proceeds to fail.

Additionally, you can make use of the <, >, and >> operators to pass outputs to and from files and commands.

Symbol What it Does
< The < operator can pipe data from one file into an executable, as demonstrated in the examples below. This allows for the automation of testing user input rather than file input. This method does not pass the file to the program, it passes the contents of the file directly into the terminal when a script requests user inputs.
> The > operator can pipe data from the executable directly into a file, rather than displaying program outputs into the terminal. This could be useful when using commands that compare program outputs to expected outputs, since piping the program output directly to a file can allow for the user to use the diff command to compare the output to a pre-existing file.
>> The >> operator acts similarly to the > operator, with the notable exception that the > will write over a file, and the >> operator will append to a file if you do not want the contents of the current file to ge overwritten.

Examples:

The following command will compile the displayFileContents.cpp file, and if the compiliation is successful, then the command will proceed to run the generated executable. Also notice the use of the < operator.

$ g++ -std=c++17 -Wall -Wshadow -Werror displayFileContents.cpp -o main && ./main < inputText
    Value 1 is: 5
    Value 2 is: 4
    Value 3 is: 7
    Value 4 is: 93
    Value 5 is: 10

If you were to use || in place of &&, it could present issues in the event that compilation fails.

$ g++ -std=c++17 -Wall -Wshadow -Werror -o main || ./main < inputText
    g++: fatal error: no input files
    compilation terminated.
    -bash: ./main: No such file or directory

Rather than displaying the program in the terminal, let's pipe the program into a separate file.

$ g++ -std=c++17 -Wall -Wshadow -Werror displayFileContents.cpp -o main && ./main < inputText > outputText
$ more outputText
    Value 1 is: 5
    Value 2 is: 4
    Value 3 is: 7
    Value 4 is: 93
    Value 5 is: 10

And rather than write over outputText, lets run the program again, but append the output to what is already in the outputText file.

$ g++ -std=c++17 -Wall -Wshadow -Werror displayFileContents.cpp -o main && ./main < inputText > outputText
$ more outputText
    Value 1 is: 5

    Value 3 is: 7
    Value 4 is: 93
    Value 5 is: 10
    Value 1 is: 5
    Value 2 is: 4
