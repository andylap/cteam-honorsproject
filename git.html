Using Git

Overview

This section will briefly go over the essential git commands you can use to interface with tools such as BitBucket, Source Forge, and most popularly, GitHub. We will discuss basic command line instructions to just get started and move on to slightly more complex commands that will be needed when working in teams, especially for the ECE 39595 final project!

Why use Git?

Simply, it makes collaboration easier. Git will track what you have added to source files, what you have deleted, allow you to revert to previous versions, and so much more. In projects with thousands of contributors, like open source projects, git is essential in making sure different people don't cause conflicts or break the current system. With engineers, it's all about efficiency, and git delivers exactly that when it comes to project management.

Git Structure

There's 4 main components to git architecture: Working directory, Staging area, Local repository, Remote repository

  • All the adding/editing/deleting you will be doing to your source files will be done in the working directory
  • Anything you want to save for your next version (A.K.A commit) of your project is in the staging area. Kind of like a rough draft.
  • Where you make local changes, probably your computer, is your local repository. Anything "committed" will be here.
  • The project is hosted "remotely" on some server in some far away place. All team members have access to this remote repository.

Git Commands

Below are some commands! Each drop down may have some additional commands that are related to the header.

$ git init

Navigate to a specified directory and enter the following command to create an empty git repository.

$ git clone https://github.com/YOUR-USERNAME/YOUR-REPOSITORY

Use the following command to create a copy of an existing repository. To clone a GitHub repository, navigate to the project page. You can either clone via HTTPS or SSH. The example above shows HTTPS. 

$ git add [filename]

To move your changes to the staging area, use the above command. Remember, these changes could also be deletions. So you can add a deleted file to the staging area. This must be done before commit

$ git add .

Don't want to add each individual file or directory? No problem, just use the above command to recursively stage all changes. 

$ git commit

Are all your changes staged? If so, use this command to take a "screenshot" of your staged changes to your local repository. You will be prompted by a text editor to enter your commit message. This should describe the changes you are committing, usually pretty short. 

$ git commit -m "[message]"

To skip the extra step of entering your commit message in an editor, use the above command to kill two birds with one stone. 

$ git push [remote] [branch]

You should probably pull before you push. The push command will sync a local branch with the remote repository. If you aren't using branches, the command will most likely look like 

$ git push origin master
since master is the default branch. "origin" specifies the name of the remote repository. 

$ git pull [remote] [branch]

This command is just a combination of git fetch and git merge. This will update your local repository with what is in the remote. Like push, if you aren't using branches this command will most likely look like 

git pull origin master
Hopefully there isn't a conflict! If you expect conflicts, it's better to first use 
git fetch
This will allow you to see what everybody else is working on without merging. 

$ git status

Shows you what's going on in your working directory and staging area. See what files are to be committed with this function. Also color coded! 

$ git log

Curious about what has been committed? Use this command to see the ids and information of all git commits.

What are branches and why? Generally you DO NOT want to work directly on the master branch especially in large projects. This will lead to inevitable merge conflicts. Often the code on the master branch is production code. Directly pushing to the master branch can potentially break a system. Instead, branches create copies that you can edit before you either merge it back into the master branch or create a pull request (developer request to integrate changes into main repository) 
$ git branch [branchname]

Creates a branch. 

$ git checkout [branchname]

Switches to a branch. 

$ git checkout -b [branchname]

Does both git branch and git checkout commands in one! 

$ git branch -a

See a list of branches.

Time to integrate changes back? First switch to the desired branch and then use the following command to merge the changes 
$ git merge [branchname]

Creates a branch.

If you are merging into master, your sequence of commands probably look like:

$ git checkout master
$ git merge branchtomerge

More likely then not, you will encounter merge conflicts at some point when doing this. There are many ways to resolve conflicts. Command Line. IDE
